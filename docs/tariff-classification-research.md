# 台灣出口到中國大陸商品，稅則如何決定

> 文件目的：整理 **台灣出口到中國大陸** 的商品，在實務上如何決定申報稅則、如何判斷 ECFA 優惠是否可能適用，以及系統模組應如何設定邊界。
>
> **重要聲明：本文件僅供 research / pre-classification / compliance design 使用，不構成最終報關、法律意見或海關裁定。最終稅則號列、稅率、ECFA 優惠是否適用，仍以中國海關接受申報、預裁定結果、正式審核與放行決定為準。**

---

## 1. 先講結論：不是先問「有沒有 ECFA」，而是先問「商品到底是什麼」

對「台灣出口到中國大陸商品，稅則如何決定」這件事，實務順序應是：

1. **先做商品稅則歸類（classification）**
   - 先確認商品本質、成分、用途、製程、包裝型態、功能。
   - 產出候選 HS / 稅則號列。
2. **再看該號列是否在 ECFA 貨品清單內**
   - 不是所有貨品都有 ECFA 早收或相關優惠。
3. **再判斷是否符合 ECFA 原產地認定基準**
   - 即使貨品在清單內，也不代表一定能享優惠。
4. **最後檢查文件與申報要件是否完整**
   - 例如原產地證明書、發票、裝箱單、成分/BOM、製程資料、合同、運輸文件等。

這四層是遞進關係，不可倒置：

- **稅則歸類** 決定你在看哪個貨品。
- **ECFA 貨品清單** 決定這個貨品有沒有被協議覆蓋。
- **原產地認定** 決定你是否有資格主張優惠。
- **文件要求** 決定你能不能把前面三件事說清楚並被海關接受。

---

## 2. 實務流程：台灣出口端 vs 中國進口端

### 2.1 台灣出口申報端在做什麼

台灣出口端的核心任務，不是替中國海關做最後裁定，而是把案件資料準備到足以支持：

1. **台灣端出口報關資料一致**
   - 商品名稱
   - 規格/型號
   - 成分或 BOM
   - 用途與產品型態
   - 出口統計號列 / 參考號列
   - 發票、裝箱單、合同
2. **ECFA 可行性初判**
   - 候選號列是否落在 ECFA 貨品清單
   - 原產地基準是否有可能符合
3. **準備原產地相關證明資料**
   - 原產地證明書
   - 製程說明
   - 原料來源資料
   - 區域價值含量（若規則要求）
4. **建立給中國進口商 / 報關行可用的申報包**
   - 讓中國進口端能用一致邏輯主張稅則與 ECFA 優惠

### 2.2 中國進口海關端在做什麼

中國進口海關端才是最終決定下列事項的一方：

1. **接受或不接受申報稅則號列**
2. **接受或不接受 ECFA 優惠主張**
3. **審查原產地文件與真實性、一致性**
4. **必要時要求補件、質疑歸類、改列他號、補稅或不予優惠**
5. **在爭議較大時，以預裁定、正式審核、實際查驗結果為準**

### 2.3 兩端必須刻意分開

系統與文件裡必須避免混淆以下兩件事：

- **台灣端可以做的：** 預分類、預判 ECFA 覆蓋、預判原產地、預檢文件缺口。
- **中國端才有權做的：** 最終進口稅則接受、優惠適用接受、海關裁量、正式審核結論。

這是合法合規邊界。任何模組都不應把「預判」寫成「確定適用」。

## 3. 四層關係：稅則歸類、ECFA 貨品清單、原產地認定、文件要求

## 3.1 第一層：稅則歸類（classification）

這是第一層，也是最重要的一層。沒有相對穩定的候選號列，後面三層都會漂移。

### 歸類時至少要看

- 商品中文 / 英文名稱
- 功能與主要用途
- 成分組成
- 各成分比例
- 是否屬複合製品
- 是否經特定製程
- 包裝型態（零售包裝、工業原料、半成品）
- 是否含特殊添加物
- 是否有規格書 / 型錄 / 成分表 / SDS / COA

### 對食品類特別重要的欄位

- 乳成分比例
- 糖含量 / 澱粉類基底
- 茶、果餡、芋頭餡、奶精、香料等構成
- 是否即飲 / 即食 / 烘焙成品 / 調製粉 / 半成品
- 單一原料 vs 混合調製品

### 模組邏輯建議

系統不應只輸出一個「唯一號列」，而應輸出：

- `candidate_hs_codes`
- `classification_basis_summary`
- `key_missing_facts`
- `classification_risk_level`

因為前期常常只能做到 **候選號列排序**，而不是海關級定論。

---

## 3.2 第二層：ECFA 貨品清單

只有當商品先有候選號列後，才能檢查：

- 該號列是否在 ECFA 相關貨品清單內
- 是否屬可主張優惠的範圍
- 是否可能有分階段降稅或年度稅率差異

### 關鍵觀念

- **ECFA 清單判斷是建立在號列之上，不是建立在商品名稱之上。**
- 同一個商品名稱，如果歸到不同號列，ECFA 適用結果可能完全不同。

因此模組設計應是：

- 先完成 `candidate_hs_codes`
- 再對每個候選號列做 `ecfa_coverage_check`
- 不能反過來從「想要優惠」去倒推號列

---

## 3.3 第三層：原產地認定

即使貨品清單有覆蓋，也還要看是否符合 ECFA 原產地規則。

### 常見判斷路徑

- 完全取得
- 非完全取得但符合實質轉型
- 稅則轉換標準
- 區域價值含量標準
- 特定製程標準

### 系統應處理的不是「一鍵判定」，而是「原產地可行性檢核」

建議輸出：

- `origin_rule_type`
- `origin_precheck_status`
- `origin_gaps`
- `required_origin_evidence`
- `high_risk_materials`

### 食品類常見風險

- 關鍵原料非台灣來源
- 主要價值成分來自第三地
- 製程僅為簡單混合、分裝、烘焙前後仍不足以構成規則要求
- BOM / 成分表 / 原料產地聲明彼此不一致

---

## 3.4 第四層：文件要求

文件層不是附屬，而是讓前面三層得以成立的證據層。

### 常見文件

- 商業發票
- 裝箱單
- 合同 / 訂單
- 產品規格書
- 成分表 / BOM
- 製程說明
- 原料來源證明
- 原產地證明書
- 運輸文件
- 進口端海關要求的補充說明

### 關鍵觀念

- **文件不是有就好，而是要彼此一致。**
- 商品名稱、規格、成分、重量、產地、型號若互相打架，會直接提高海關質疑風險。

因此模組應輸出：

- `document_requirements`
- `missing_documents`
- `consistency_warnings`

## 4. 建議的實務判定步驟（可直接當模組流程）

## Step 1：建立商品事實檔（product facts）

先收集：

- 商品名稱（中 / 英）
- 規格型號
- 用途
- 成分與比例
- 製程
- 包裝型態
- 交易條件
- 目的地（中國大陸）
- 製造地 / 原料來源地

若這一步資料不足，系統不應硬判。

## Step 2：做候選稅則歸類

依據商品事實檔，列出：

- 候選 4 碼 / 6 碼 / 8 碼 / 10 碼號列
- 每個候選號列的判定理由
- 哪些事實會導致號列改變

## Step 3：切分台灣端與中國端視角

### 台灣端
- 對應出口統計 / 報關使用的參考號列
- 整理可提供給進口商的 supporting facts

### 中國端
- 對應進口申報號列
- 檢查該號列是否在 ECFA 清單
- 檢查是否需預裁定 / 進一步諮詢

## Step 4：檢查 ECFA 貨品清單覆蓋

對每個候選號列做：

- `covered`
- `not_covered`
- `uncertain_need_manual_review`

## Step 5：檢查原產地規則可行性

至少問：

- 是否完全取得？
- 若否，是否符合稅則轉換？
- 是否符合區域價值含量？
- 是否符合特定製程？
- 缺哪些原料來源 / 成本 / 製程資料？

## Step 6：檢查文件完備度

- 原產地證明書是否可申請
- 文件欄位是否一致
- 是否已有足夠資料支持 ECFA 主張

## Step 7：輸出風險結論，但保留海關邊界

建議只輸出下列等級，而非假裝最終答案：

- `preliminarily_supported`
- `possible_but_needs_more_evidence`
- `high_risk_of_reclassification`
- `not_ready_for_ecfa_claim`
- `should_seek_customs_ruling`

## 5. 模組判定標準：官方資料來源與連結

下列來源應視為模組規則的優先官方依據。

## 5.1 台灣端官方來源（已可確認）

### A. 財政部關務署－原產地規則總頁
- 連結：https://web.customs.gov.tw/singlehtml/715
- 用途：作為 ECFA / FTA 原產地規則官方入口。

### B. 海峽兩岸經濟合作架構協定（ECFA）原產地認定基準
- 連結：https://web.customs.gov.tw/download/m232104
- 用途：判斷原產地基準、實質轉型、區域價值含量等。

### C. 海峽兩岸經濟合作架構協定（ECFA）原產地貨品清單
- 連結：https://web.customs.gov.tw/download/m231959
- 用途：檢查候選號列是否在 ECFA 貨品覆蓋範圍。

### D. 海峽兩岸經濟合作架構協定（ECFA）統一規章（含原產地證明書）
- 連結：https://web.customs.gov.tw/download/m231957
- 用途：檢查原產地證明書與相關作業規範。

### E. 財政部關務署－稅則稅率查詢及諮詢電話（官方入口）
- 入口頁：https://web.customs.gov.tw/link/3078
- 轉址查詢系統：https://portal.sw.nat.gov.tw/APGQ/XGC411
- 用途：作為台灣端官方稅則 / 稅率查詢入口與人工諮詢入口。

## 5.2 中國進口端官方來源（模組應引用，但結論須保守）

> 備註：中國海關官方網站對自動化請求有明顯 anti-bot / WAF 限制；實作上應保留人工打開官網核驗流程，不建議在 MVP 階段把爬取當成穩定資料源。

### F. 中華人民共和國海關總署官方網站
- 連結：https://www.customs.gov.cn/
- 用途：中國海關法規、公告、進口申報與歸類制度的官方入口。

### G. 海關總署「預裁定」制度官方入口（應由人工於官網查最新規定）
- 官方網站入口：https://www.customs.gov.cn/
- 用途：當商品歸類、原產地或申報要素存在重大不確定性時，應評估是否走中國海關預裁定 / 正式裁定機制。

### H. 中國年度《進出口稅則》與相關公告（應使用當年度官方版本）
- 官方網站入口：https://www.customs.gov.cn/
- 用途：確認中國進口端當年度有效稅則、最惠國稅率、協定稅率、暫定稅率與調整公告。

### I. 中國進口申報與單一窗口 / 海關線上系統（作為實務申報入口）
- 官方網站入口：https://online.customs.gov.cn/
- 用途：進口申報、企業端作業與部分官方服務入口。

### J. 中國政府法規公開平台（作為海關行政法規 / 規章補強來源）
- 入口：https://www.gov.cn/
- 用途：補查海關行政法規、部門規章與年度公告的正式公開版本。

### 中國端資料使用原則

模組若引用中國端資料，應強制加註：

1. **以申報進口當年度有效版本為準**
2. **以中國海關實際接受之號列與稅率為準**
3. **若有歧義，應建議預裁定 / 正式諮詢**
4. **MVP 不應把中國端網站資料快取後當成永久正確答案**

## 6. 合法合規邊界：哪些能自動判，哪些不能

## 6.1 可以由模組做的

- 候選號列排序
- 缺資料提示
- ECFA 清單覆蓋初判
- 原產地規則可行性 precheck
- 文件完備度檢查
- 風險與人工覆核建議

## 6.2 不應由模組假裝做到的

- 最終海關歸類裁定
- 最終適用稅率保證
- 最終 ECFA 優惠適用保證
- 替代中國海關預裁定
- 在資料不足下輸出單一肯定結論

## 7. 對現有 ECFA 專案的建議串接方式

建議把「稅則判定模組」放在現有 `ecfa_precheck` 之前，形成下列流程：

1. `product_facts_intake`
2. `classification_candidates`
3. `ecfa_coverage_check`
4. `origin_precheck`
5. `document_check`
6. `optimization`（僅在前面已有基本可分析性時才進入）

### 建議輸入欄位

- product_name
- product_description
- intended_use
- ingredients / bom_items
- manufacturing_process
- packaging_form
- export_origin_country
- destination_country
- supporting_documents_present

### 建議輸出欄位

- candidate_hs_codes
- selected_working_hs_code
- classification_basis_summary
- ecfa_coverage_status
- origin_precheck_status
- document_requirements
- missing_fields
- warnings
- recommended_next_checks
- customs_boundary_notice

### 必加欄位：海關邊界聲明

建議固定輸出：

```json
{
  "customs_boundary_notice": "This result is a pre-classification and ECFA eligibility precheck only. Final tariff classification, duty treatment, and preferential eligibility remain subject to acceptance, review, or advance ruling by the competent China Customs authority."
}
```

## 8. 給 Shaun 驗證合法合規時可直接看的重點

如果 Shaun 要驗證這份設計是否夠保守、夠合法合規，建議直接看這 5 點：

1. **有沒有清楚區分台灣出口端與中國進口海關端？**
2. **有沒有把稅則歸類放在 ECFA 判斷前面？**
3. **有沒有把 ECFA 清單、原產地規則、文件要求拆成四層？**
4. **有沒有只做 precheck，而沒有冒充最終裁定？**
5. **有沒有把最終接受權限明確留給中國海關 / 預裁定 / 正式審核？**

如果以上 5 點都做到，這個模組就比較接近「可用於業務前置判斷」而不是「高風險誤導工具」。

---

## 9. 最後結論

對台灣出口中國大陸商品而言，稅則決定不是單一步驟，而是：

- **先分類**
- **再看 ECFA 清單**
- **再看原產地規則**
- **再看文件證據**
- **最後由中國海關接受申報 / 預裁定 / 正式審核定案**

因此，系統最合理的定位不是「自動給最終答案」，而是：

**用官方資料做可追溯的 pre-classification 與 ECFA precheck，幫案件更快進入人工覆核與正式申報。**
